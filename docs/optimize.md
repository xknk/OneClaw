# AI Agent 系统架构优化技术方案

## 1. 任务内部调度：工具执行的并行与串行  - 初步完成
**核心点**：最大化单次请求的执行效率。
- **并行 (Parallel)**：针对无副作用的工具（`read_file`, `grep`, `mcp_get`），使用 `Promise.all` 批量执行。
- **串行 (Sequential)**：针对有环境破坏性的工具（`write_file`, `exec`, `git`），强制排队执行，防止文件冲突。
- **实现**：在工具定义中增加 `isReadOnly` 属性，供执行引擎动态分流。

---

## 2. 安全护栏：敏感操作的人工确认 (HITL)
**核心点**：防止 AI 产生不可逆的破坏行为。
- **拦截**：定义敏感工具清单（如 `rm`, `delete_file`）。
- **交互**：当触发敏感工具时，Generator `yield` 出一个确认请求，后端挂起等待前端回传指令。

---

## 3. 成本控制：上下文微压缩与缓存
**核心点**：死磕 Token 消耗，对齐 API 缓存。
- **微压缩**：正则清理 ANSI 颜色代码、冗余空行；路径由绝对路径转为相对路径 - 已完成。
- **缓存**：内存 LRU 缓存高频文件内容，减少磁盘 I/O。
- **固化**：保持 System Prompt 字节级不变，确保命中厂商的 Prompt Cache - 已完成。

---

## 4. 交互体验：响应式前端与全链路中止
**核心点**：让 Agent 运行状态透明、可控。
- **Abort 穿透**：点击停止时，通过 `AbortController` 同时杀掉 AI 请求和本地子进程（如正在运行的 `npm install`）。
- **实时看板**：通过生成器实时输出 Thinking、Token 消耗、工具执行进度。

---

## 5. Token 预算：动态工具加载与结果采样
**核心点**：防止长对话导致的上下文溢出（Context Overflow）。
- **动态加载**：初始仅传入基础工具，高级工具（MCP/插件）按需动态注入。
- **结果采样**：对超长工具返回（如几千行日志）执行“头尾保留+中间摘要”处理。

---

## 6. 架构稳定性：多任务隔离与 Context 重构
**核心点**：支持多入口任务同时运行，消除全局变量隐患。
- **禁止全局变量**：严禁将 `callCount`（调用计数）设为全局变量应该做为参数传入。
- **任务上下文隔离**：每个 `query` 生成器实例创建时，分配独立的 `TaskSession` 对象。
- **独立配额**：确保任务 A 的工具调用次数不会消耗任务 B 的配额，实现完全的逻辑隔离。

## 7、**流式输出**
#### 核心思路：不要等所有内容都准备好才显示，而是“边生成边显示”。
#### 技术：利用 Stream API。
#### 做法：当你生成一段内容时，就立即显示出来，而不是等所有内容都生成完了再显示。
#### 结果：用户可以实时看到结果，而不是等所有内容都生成完了再显示。

## 9、**流式工具处理**
#### 核心思路：流式处理信息并调用工具，当一个工具的json体完整后可立即调用此工具（前提是当前工具是独立的），判断其是否可以并行调用工具，如果可以，则立即调用，反之，等上一个工具调用完成才可调用。
#### 技术：利用generator的yield机制，当一个工具的json体完整后可立即调用此工具（前提是当前工具是独立的），判断其是否可以并行调用工具，如果可以，则立即调用，反之，等上一个工具调用完成才可调用。
#### 做法：增加标识位，判断工具是否完整、是否可执行（独立的，且该工具的输入不是上一个工具的输出），如果完整且可执行则立即调用。
#### 结果：实现快速工具调用，而不是等所有信息输出后统一调用。

## 10、**配置轻模型入口-供降级、上下文摘要选择**
#### 核心思路：提供轻量级模型入口，供降级、上下文摘要选择。
#### 技术：增加轻量级模型入口，可动态配置。
#### 做法：当上下文过长时，可利用轻量级模型进行摘要。当大模型出错时，可利用轻量级模型进行降级。
#### 结果：提供轻量级模型入口，供降级、上下文摘要选择。

2、**三阶段启动模型 (Three-Phase Startup)**
#### 核心思路：不要等所有代码都准备好才动，而是“分批行动”。
#### 副作用阶段：只执行最轻量的逻辑（如读取简单的环境变量）。
#### 导入阶段：加载必要的模块。
#### CLI 分发阶段：根据你的命令（是想聊天还是只想看版本号）决定最后执行什么。
#### 意义：通过这种拆分，程序可以最早地给出反馈，而不是卡在那里黑屏加载。

3、**增加并行加载模式**
#### 背景：在 JS 中，import 很大的库（比如 React 或庞大的 SDK）是耗时的同步过程，会卡住 CPU。
#### 骚操作：Claude Code 在执行第一行 import 之前，先偷偷启动一个子进程（比如去联网、去读 Key）。当你那些库加载完了，子进程的活也干完了。
#### 结果：原本需要“加载耗时 + 子进程耗时”，现在变成了“两者取其大”，实现了所谓的“零成本”启动。

4、**编译期特性标志**

#### 核心思路：代码“瘦身”，内外有别。
#### 技术：利用 Bun 编译器的 feature() 宏。
#### 做法：如果你是外部用户，编译器在打包时会自动把内部开发调试用的代码、测试代码物理删除（Dead Code Elimination）。
#### 结果：你下载的二进制包里一个字节的废代码都没有，文件更小，加载更快。

5、**快速路径设计 (Fast-path Design)**
#### 核心思路：小事绝不动干戈。
#### 情景：如果你只想运行 claude --version，程序不应该去初始化 AI 环境、不应该联网、不应该加载复杂的 Prompt 模板。
#### 做法：为简单指令设计专门的“逃生出口”，只加载 1% 的代码就立即返回结果。
